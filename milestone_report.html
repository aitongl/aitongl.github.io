Work Completed So Far

We started off by understanding how the Glucose library works. We identified the relevant sections of the library and implemented a purely sequential MaxSAT solver. It iterates B = 0, 1, 2, ... and for each B, encodes the constraint that sum(relax_vars) ≤ B and calls Glucose's solver with this assumption. Learned clauses are reused across iterations, giving the sequential solver a significant advantage over solving from scratch each time.

We also experimented with a naive parallel version that splits the search space by partitioning the minimum number of soft clause violations. After calibration, we switched to a dynamic work queue approach that assigns the next unclaimed bound B to whichever thread finishes first, avoiding load imbalance from static partitioning. Threads share only the current best cost atomically, allowing early termination when another thread finds a better solution.

Progress vs. Proposal

We are on track with respect to our core goals. The first iteration of our algorithm is working, however it is not performing well across multiple cores. We suspect this will slow future progress and may hinder our ability to expand to multiple communication protocols and other extended goals. Concerns

  • The benchmark test suite is too large in terms of memory. We need to randomly select a subset.
  • There is significant variation in how long each test takes. Some tests produce correct results in 0.001s even with a sequential approach. We need to focus on harder instances where individual SAT calls are expensive and the optimal cost is large enough that multiple B values need to be explored.
  • The parallel version currently shows no speedup on tested instances. The main bottleneck is that each thread builds its own independent solver and counter table from scratch.
  • We are also unsure whether more clause sharing across cores will provide a significantly better parallel algorithm in terms of speedup.
